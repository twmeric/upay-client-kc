# 母机团队守则 - 部署回滚规范

## 📋 事件总结

**日期**: 2026-03-20  
**项目**: King-Chicken 支付系统  
**问题**: 部署错误后回滚困难，导致生产环境长时间不可用

---

## ❌ 问题分析

### 1. 根本原因

#### A. 缺乏版本控制
- **问题**: 没有使用 Git 进行代码版本控制
- **后果**: 无法快速回滚到上一个稳定版本
- **教训**: 所有代码必须纳入版本控制

#### B. 部署流程混乱
- **问题**: 多个项目共用同一个 Cloudflare Pages 项目 (`payment-portal`)
- **后果**: 不同版本互相覆盖，无法追溯历史部署
- **教训**: 每个环境应该使用独立的项目名称

#### C. 缺乏部署记录
- **问题**: 没有记录每次部署的版本号和变更内容
- **后果**: 回滚时不知道应该回滚到哪个版本
- **教训**: 每次部署必须记录版本号和变更日志

#### D. 本地构建产物和线上不一致
- **问题**: 本地 `dist` 目录被覆盖，无法重建历史版本
- **后果**: 只能通过下载线上版本进行恢复
- **教训**: 构建产物应该作为 artifacts 保存

## ✅ 预防措施

### 1. 版本控制规范

```bash
# 必须使用 Git
- 每个项目初始化 Git 仓库
- 每次部署前必须提交代码
- 使用有意义的 commit message
- 打标签标记生产版本: git tag -a v1.2.3 -m "Production release"
```

### 2. 部署规范

#### A. 环境分离
```
开发环境: payment-portal-dev
测试环境: payment-portal-staging  
生产环境: payment-portal-prod
```

#### B. 部署前检查清单
```markdown
□ 代码已提交到 Git
□ 版本号已更新
□ 变更日志已记录
□ 回滚方案已准备
□ 部署时间已记录
```

#### C. 部署记录模板
```markdown
## 部署记录

**版本**: v1.2.3
**时间**: 2026-03-20 14:30:00
**部署人**: [姓名]
**变更内容**:
- 新增功能 X
- 修复问题 Y

**回滚版本**: v1.2.2 (Deployment ID: xxx)
**验证结果**: [ ] 通过 [ ] 失败
```

### 3. 回滚方案

#### A. 快速回滚步骤
```bash
# 1. 立即回滚 DNS (30秒内完成)
# 2. 清除 Cloudflare 缓存
# 3. 验证回滚结果
# 4. 记录回滚原因
```

#### B. 回滚记录
```markdown
## 回滚记录

**时间**: 2026-03-20 15:00:00
**原因**: 新功能导致白屏
**回滚版本**: v1.2.2
**回滚人**: [姓名]
**恢复时间**: 15:05:00
**后续行动**: 修复白屏问题后重新部署
```

### 4. 构建产物管理

#### A. 保留历史构建
```
builds/
├── v1.2.3/          # 保留每个版本的构建
├── v1.2.2/
├── v1.2.1/
└── current -> v1.2.3/  # 软链接指向当前版本
```

#### B. 构建信息文件
```json
{
  "version": "1.2.3",
  "buildTime": "2026-03-20T14:30:00Z",
  "gitCommit": "abc123",
  "deployedBy": "[姓名]"
}
```

## 🚨 紧急回滚流程

### 情况: 生产环境出现故障，需要立即回滚

```bash
# Step 1: 确认回滚版本 (30秒)
# 查看部署记录，确认上一个稳定版本

# Step 2: 更新 DNS (30秒)
# 使用 DNS_API_TOKEN 更新 Cloudflare DNS

# Step 3: 清除缓存 (10秒)
curl -X POST "https://api.cloudflare.com/client/v4/zones/$ZONE_ID/purge_cache" \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"purge_everything":true}'

# Step 4: 验证 (30秒)
# 检查网站是否恢复正常

# Step 5: 记录 (2分钟)
# 填写回滚记录，通知相关人员

# 总用时: 3-5 分钟
```

## 📚 经验教训

### 本次事件总结

1. **不要在生产环境直接测试**
   - 先在开发/测试环境验证
   - 使用 feature flags 控制新功能发布

2. **保留多个备份**
   - 本地备份 + 云端备份
   - 至少保留最近 5 个稳定版本

3. **快速响应**
   - 发现问题立即回滚，不要试图在线修复
   - 先恢复服务，再排查问题

4. **团队协作**
   - 部署前通知相关人员
   - 保留部署和回滚的决策记录
